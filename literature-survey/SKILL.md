# 深度文献调研 (Literature Survey)

对一个研究方向做系统性文献调研，写一份领域新人能读懂的深度综述报告。报告用中文写，技术术语保留英文。读者是**还不熟悉该领域的人**：术语、缩写、任务设定第一次出现时要解释清楚。


## 范围与筛选标准

- **时间窗口**：近五年，重点看最近一两年（如果用户指定了范围，请遵循用户指令）。
- **核心来源**：CCF-A 类会议与期刊。判级查 `references/ccf-2026-lite.md`，按**全称**核对。排除 workshop / demo / short paper，影响力突出者除外。交叉领域主题（航天、能源、生医等）不要把 CCF 硬套过去，用 paper-search 查期刊指标，取 JCR Q1 或中科院 1、2 区。
- **补充来源**：不在 CCF-A 但满足任一条件即纳入，报告中注明依据：
  - GitHub star ≥ 100（`gh api` 或 web 查询，以查询当日为准）；
  - 引用量 ≥ 100（引用量以 Semantic Scholar 或 OpenAlex 的计数为准：`search_semantic_scholar` 和 `search_openalex` 的检索结果自带引用数，单篇论文用 `get_paper_by_doi` 查）；
  - 一年内的预印本：重点关注知名作者/团队的新作（既往有 CCF-A 或高影响力发表记录），以及 OpenReview 投稿 CCF-A 会议且评分不低、被重要工作引用、引用增速快、代码仓库热度高的工作。

## 工具

- 检索、元数据核验、施引/参考文献扩展、PDF 获取一律使用 paper-search skill；用户同意后才退化为 web 检索，并说明元数据可靠性会下降。
- 读全文优先取 **LaTeX 源 > Markdown / HTML > PDF**：arXiv 论文先拿源码包（`https://arxiv.org/e-print/<id>`，tar 解包读 .tex）或 ar5iv 页面（`https://ar5iv.org/abs/<id>`）；都拿不到才下载 PDF 解析。

## 工作流

1. **范围界定**：划清研究问题边界（拿不准主题边界时，请一定询问用户，确保和用户达成共识）；列关键词及同义词（例：content moderation ≈ safety filtering ≈ NSFW detection，同一问题常有几套术语，只搜一种会漏掉半个领域）。
2. **广撒网初筛**：paper-search 多来源、多措辞检索，再对核心 venue 近两年 proceedings 定向扫一遍（DBLP 等来源）。这一步只看标题、摘要、venue、年份、引用数，建 **30–60 篇**候选清单。用工作表记下题名、作者、年份、venue、DOI/arXiv ID、引用数、初步相关性。作者名和 DOI 从检索记录里带走，写作时不要靠记忆补：记忆会补出貌似合理但错误的人名。
3. **判级与筛选**：按上节标准过滤；走补充来源通道的逐条核验并记下依据；近一年 arXiv 新作归入前沿动态候选。
4. **引文图谱扩展**：从初步结果出发，查找前向引用（谁引用了这篇论文？）和后向引用（本文引用了哪些文献？），新发现按同样标准并入。在新论文里见到没搜过的术语，拿它回到第 2 步再检索一轮。
5. **开源发现**：GitHub 搜主题相关实现，Papers With Code 查对应任务/数据集的模型与代码，再找社区维护的 awesome-X 清单。结果喂给第 5 章开源生态和论文总表的代码链接列；从热门项目反查到的论文也按筛选标准并入候选。
6. **精读论文**：对通过筛选的读全文，逐篇记录动机、方法核心、实验亮点、局限、与前后工作的关系、完整英文摘要原文。
7. **Gap 识别与验证**：用覆盖矩阵找 gap——问题维度做行、现有方法做列，空格就是候选。每个候选逐条验证：换 3 种以上措辞检索、翻 top venue 近两年 proceedings、查近半年 arXiv、技术上可行、填上它对领域确实有意义。全过才算 gap，置信度标高/中/低（高 = 充分检索、明确没人做；低 = 检索有限，可能漏在别处）。
8. **成文**：按"输出文档"组织。报告里每个事实性字段（DOI、venue、年份、作者、star 数、实验数字）都要能追溯到检索记录或论文原文。

## 验证规则

- 不凭记忆编造 DOI、arXiv ID、venue、年份、作者，写进报告前经 paper-search 核验。题名用论文的印刷标题（系统名放括号里，不拿它当标题），venue 以官方记录为准；题名、一作、年份、DOI 相互对不上就标记可疑重查。
- 同一工作的 arXiv 版和正式发表版合并成一条，优先正式版本。交付前对所有标 preprint 的条目做一次正式版本对账（查 Semantic Scholar/OpenAlex 的发表 venue）。arXiv 论文常常早已中会，标错会让读者低估它的份量。
- 摘要原文和官方摘要逐字一致（只允许空白差异）。拿不到完整摘要就标"摘要节选"或"未能核验"，不编造，不截断冒充完整。一次 403 不等于取不到，换 UA、换来源（官方页 / arXiv / OpenAlex）再试。
- 高热度或反直觉的结论查撤稿状态（Crossref 的 update-to 字段）。用户要找"证明 X 的那篇论文"而 X 已被推翻或从未成立时，如实说明该主张的现状，不递上最接近的引用。
- gap 结论写明检索来源和日期（"截至 X 日，在 Y 来源 Z 检索式下未检出"）。检索源限流或报错时的零结果，不能当"无人做过"的证据。
- 有争议的问题呈现双方证据；置信度跟着证据强度走：单篇报告的结果写"某工作报告了 X"，多方复现的结论才平铺直叙。

## 写作原则

- **先检索后写作**：记忆负责框架和检索词，检索结果负责引用和事实。
- **综述是比较，不是摘要罗列**：说清哪些结论互相印证、哪些矛盾、谁在什么设定下取代了谁。技术脉络用连贯散文写成论证，每段首句是你的综合判断，引用跟在后面。自检办法：只读每段首句，连起来应该是一条论证链；如果是一串人名年份，那是套着综述外衣的注释书目，重写。讲清为什么从 A 方法走到 B 方法，A 的哪个局限催生了 B，而不是把两者并列。"A 催生 B"要有 B 引用 A 的证据，并过一遍年代检查（后发工作"支撑"不了先发工作）；拿不准就写"同期工作"或"概念相关"，时间挨着或同一个团队不等于因果。
- 内联引用写成 `[Author Year](https://doi.org/10.xxxx/xxx)`，arXiv 没有 DOI 就链 abs 页。不用 `[1][2][3]` 编号，段落一挪就错位。DOI 里的括号转义成 `%28` `%29`。

## 输出文档

完整报告共六章：

1. **研究背景与问题定义**：1–2 段说清该领域要解决什么问题。如果开工前没条件和用户确认主题边界（比如在后台运行），把自己定的范围假设列在本章开头，不超过 5 条，读者一眼就能判断报告口径对不对。
2. **技术脉络梳理**：按子方向或时间线组织的综述散文加内联引用，讲明白方法怎么演进，关键节点展开原理和动机。章末给一屏"技术地图"，图或表都行，关系必须带类型标签（改进自 / 同期 / 同团队后续 / 概念相关），不画无标签箭头。本章可以引用前沿动态里的 preprint，标注即可。
3. **重要工作详述**：章首先给**推荐阅读顺序**——5–8 篇新人最短路径，每篇一句为什么先读它；然后按脉络顺序逐篇详述，值得推荐的都收，每篇含：
   - 基本信息：标题、作者、发表场所、年份、GitHub 链接与 star 数（如有）；非 CCF-A 来源附达标依据
   - 动机、方法（1–2 段展开核心思路和技术细节，不是一句话概括）、实验亮点、局限、与前后工作的关系
   - 完整英文摘要原文及中文翻译（原文逐字，见验证规则；译注放方括号；摘要本身是中文无需翻译）
   - 推荐理由
4. **前沿动态观察**：首次公开不满 12 个月、尚无正式发表版本的值得关注预印本。不设数量上限，不占第 3 章名额。逐篇标注"未同行评审"和选入理由（见筛选标准）；已被 CCF-A 录用的移进第 3 章。
5. **开源生态分析**：有影响力的开源项目，给 star 数、维护活跃度、适用场景。
6. **开放问题与趋势**：具体的 gap 和新方向，每个 gap 标置信度、写明检索范围。

报告末尾附**论文总表**（附录）：通过筛选的论文全部进表，按技术路线分组。列：题名、年份、venue 与判级依据、引用数、代码链接、简要概述（几句话说清这篇做了什么、特别之处在哪）。

**交付形态**：全文写入 `survey-<主题slug>-<YYYYMMDD>.md`，默认放当前工作目录，用户指定路径时从其指定。默认唯一的文件交付物就是这份报告，工作表、笔记、下载物等中间工件留在临时目录。聊天回复给实质性摘要：领域核心问题、脉络主线、3–5 篇最该先读的论文和理由、主要开放问题，文件链接放末尾。不主动产出 BibTeX 或 HTML/PDF，用户要求时另行处理。
